# 前端风格化说明

本文档用于补充 `docs/DEVELOPER/front.md`，重点说明当前白石前端的视觉语言、主题规则、组件风格约束与页面级设计一致性要求。它描述的是“页面应该长什么样”，而不是“目录里有什么文件”。

## 1. 总体风格定位

当前前端采用“Warp 深色结构纪律 + 水墨画视觉语义”的混合方向，落地为一个偏桌面创作工具气质的多页面原生前端。

整体感受应同时满足以下四点：

- **工具感**：页面结构清晰，模块边界明确，像可实际操作的创作工具，而不是营销页。
- **纸墨感**：颜色、留白、边框、投影和纹理都应带有宣纸、墨色、朱砂印章的东方视觉语义。
- **克制感**：不要使用大面积高饱和渐变、夸张光效、霓虹描边或花哨动画。
- **统一感**：所有页面都应共享同一套壳层、圆角矩形外框、按钮基线、toast 风格和主题切换规则。

项目当前不是纯网页产品，也不是移动优先页面，而是**桌面应用式 Web UI**。因此设计上优先强调：

- 左侧导航 + 主内容工作区
- 稳定的窗口感边界
- 可读、可编辑、可审阅的结果面板
- 对图片、文案、设置、历史等模块保持一致的交互骨架

## 2. 视觉语言

## 2.1 色彩基调

核心色彩体系定义在 `front/css/ink-wash.css`，当前主视觉不是冷黑科技风，而是带暖意的纸墨中性色。

主要视觉角色如下：

- `--bg`：页面底色，承担整体宣纸/夜墨背景
- `--surface` / `--surface-2`：卡片、面板、弹层、输入区的承载底
- `--fg`：主文字颜色
- `--muted` / `--meta`：说明、弱化信息、字段标签
- `--accent`：强调色，当前主要承担朱砂印章、关键按钮、少量选中态
- `--border`：边框与分隔线，必须克制、柔和，不能过硬

使用规则：

- 强调色只用于少数关键操作、印章元素、局部选中态，不可泛滥。
- 不要额外引入新的品牌色体系，除非先同步到全局设计令牌。
- 页面层级主要通过 `surface`、边框、留白和排版建立，不依赖重阴影或强发光。

## 2.2 字体与文字气质

当前字体系统也是混合设计：

- 正文 / UI：系统无衬线与现有变量定义的正文族
- 标题：更强调结构感与屏幕阅读清晰度
- 点缀：毛笔/书法风格字体，仅用于局部氛围强化
- 数值 / 元数据：等宽字体，用于参数、状态、标签、统计

使用约束：

- 毛笔字体只能作为点缀，不能整页滥用。
- 大段正文、表单、参数区、按钮文本，必须保持清晰、现代、可读。
- 元数据、参数、时间、模型信息优先使用 `mono` 语义风格。
- 不要使用花哨的英文展示字体破坏整体统一性。

## 2.3 形状与边框

当前页面外观的关键识别点之一是：**全局最外层窗口为圆角矩形外框**。

具体约束：

- 应用内页统一使用共享壳层 `.window > .app`。
- 最外层需要有明确圆角、边框和适度阴影，形成“桌面软件窗口”感。
- 页面内部卡片圆角应与全局壳层形成层级，不可所有元素都使用同一个大圆角。
- 图片预览、历史卡片、结果卡片、设置面板等都要服从统一圆角语言。

不推荐：

- 突兀的胶囊形大按钮到处出现
- 过多悬浮玻璃态
- 过深阴影造成“卡片漂浮”感
- 圆角尺度混乱

## 2.4 留白与密度

白石当前是**中密度桌面 UI**，不是稀疏型展示页，也不是极密后台表格。

应保持：

- 模块之间有明确呼吸空间
- 卡片之间有足够 gap，不能挤成瀑布流式拥堵布局
- 表单行距、字段间距和操作区层级清楚
- 历史卡片、风格卡片、结果卡片尺寸统一，不因图片比例导致整体视觉散乱

尤其要继续遵守当前项目中已形成的实践：

- 卡片间距不宜过紧
- 预览图在卡片内必须铺满卡片，不留白边
- 卡片图片需要与卡片本身保持一致圆角
- 详情预览层再按原图比例展示真实内容

## 3. 主题模式规则

项目支持浅色 / 深色双主题，依赖 `document.documentElement[data-theme]` 切换。

所有新样式都必须遵守以下原则：

- 默认样式优先使用现有 CSS 变量，不要写死浅色或深色值。
- 弹层、预览、toast、对话框、详情面板必须跟随当前主题。
- 若某组件需要单独 dark 覆盖，只允许写局部 `[data-theme="dark"]` 规则，不要再造一套主题系统。
- 图片详情弹层、历史预览层、文案详情层等覆盖层，必须与所在页面主题一致。

已经明确验证过的要求包括：

- 历史作品大图预览必须与当前主题模式一致
- 卡片打开的详情弹层不得出现和页面相反的明暗模式

## 4. 页面壳与布局骨架

## 4.1 页面级统一骨架

应用内页普遍采用如下结构：

- 左侧主导航
- 顶部品牌/标题或局部工具栏
- 中央主工作区
- 局部侧栏 / 面板 / 结果区
- 右下角共享操作提示

设计要求：

- 左侧导航负责全局切换，不承担复杂业务交互。
- 主工作区负责当前任务本身，层级必须明显高于装饰内容。
- 页面之间可以有不同模块排布，但不能失去统一壳感。
- 首页虽然不是功能页，也要和应用内页共享外层边框与整体调性。

## 4.2 工作区风格

工作区应优先强调“创作流程可操作”，常见组合有：

- 左输入 / 右结果
- 上筛选 / 下卡片列表
- 左列表 / 右详情
- 主结果区 + 辅助参数区

不要把工作区做成纯展示海报，也不要塞入设计器控制面板式的调参 UI 噪音。

## 5. 共享组件风格约束

## 5.1 按钮

按钮风格由 `app-chrome.css` 提供共享基线，应保持以下区分：

- 主按钮：承担关键操作，如生成、保存、提交
- 次按钮：辅助操作，如重置、刷新、检查更新
- 危险按钮：删除、批量删除等高风险行为
- 幽灵按钮 / 图标按钮：局部轻操作

约束：

- 不要在单页内部重新定义一套完全不同的按钮语义。
- 不要让主按钮过度发光、过亮或尺寸失控。
- 图标按钮应小而稳，不可与删除按钮位置冲突。
- 收藏、删除、多选等右上角操作需要明确避让，不能叠位。

## 5.2 Toast 操作提示

右下角操作提示已经统一收敛到共享实现，当前属于全局基础设施。

规范如下：

- 所有页面统一调用 `window.BaishiShared.toast(...)`
- 不允许页面私自维护第二套 `toast()` 动画和 DOM 结构
- 同屏数量有限制，避免连发堆叠
- 动画以轻量平移淡入淡出为主，不使用夸张缩放
- 文案应短、清楚、能读完

已形成的项目规则：

- `settings`、`history` 等页面不得再做放大版局部 toast 覆盖
- “检查更新”等长文本提示可延长停留时间，但不能做成大块遮挡

## 5.3 弹层 / 详情预览

当前项目里的弹层主要包括：

- 图片大图预览
- 文案全文阅读
- 历史详情弹层
- 设置中的局部说明面板

风格要求：

- 弹层背景、边框、文字必须跟随主题
- 关闭方式明确：关闭按钮、遮罩点击、`Esc`
- 图片详情里应展示提示词与参数
- 文案详情里应展示完整文本和原提示词
- 竖图/横图详情布局应根据原图比例自适应

## 5.4 卡片

项目当前有多种卡片：

- 风格卡片
- 历史图片卡片
- 历史文案卡片
- 结果预览卡片
- 首页入口卡片

统一要求：

- 卡片尺寸体系稳定，不随内容大幅抖动
- 预览图在卡片内统一裁切铺满
- 卡片点击区域清晰，hover / focus 态轻量但可感知
- 卡片上的收藏、删除、多选、放大等控件不能互相冲突
- 管理态与浏览态的行为必须清晰分离

## 5.5 表单与设置项

设置页是高频配置区，表单风格不能漂移成营销页组件。

要求：

- 输入框、下拉、请求体编辑区都走统一输入体系
- 供应商切换后的“获取 API”链接要明显，但仍属于表单说明层级
- 供应商名称应显示真实名称，链接应指向真实官网/文档
- API Key、请求体、模型等字段要优先保证清晰与可读

## 6. 重点页面的风格化要求

## 6.1 文生图

`text-to-image.html` 当前的视觉重点是“创作输入 + 结果审阅”。

要求：

- 固定负面提示词区域要像辅助工具，不要抢主提示词主体层级
- 结果卡片尺寸统一，点击后再看原图
- 结果区应突出“可浏览”和“可下载”，而不是杂乱堆图
- 加载、空状态、结果态要切换清楚

## 6.2 妙笔生花 / 文生文

`copywriting.html` 当前已经具备“审阅视图 / 编辑视图”双模式。

要求：

- 默认进入审阅视图，强调可读性
- 编辑视图强调可编辑性，不做 markdown 渲染
- 两个视图共享同一份内容源，切换手感平稳
- 审阅态应像最终稿阅读区，编辑态应像工作草稿区

## 6.3 历史作品

`history.html` 是当前最复杂的混合内容页之一。

要求：

- 图片与文案可以共存，但筛选与类型必须清晰
- 收藏是正式分类，不是临时状态
- 图片卡片点击后必须能查看大图、提示词和参数
- 文案卡片点击后必须能阅读完整文案和原提示词
- 管理模式未触发时，不显示批量条和多选框

## 6.4 设置 / 关于

`settings.html` 是配置中心，不是宣传页。

要求：

- 模块分组清晰，优先可配置性和可读性
- “关于”是设置页内的正式面板，不另起独立落地页风格
- 开源说明、项目说明、更新功能属于说明型模块，应稳重、克制
- “检查更新”一类反馈必须使用小尺寸可读 toast，而不是放大提示

## 7. 不应再出现的风格问题

以下问题已经在项目推进过程中被反复修正，后续不应回归：

- 页面切换使用夸张过渡动画
- 某个页面重新造一套更大、更快的 toast
- 卡片预览尺寸不统一
- 浅色模式页面里弹出深色预览层
- 收藏按钮和删除按钮叠位
- 未进入管理态时显示多选框或批量操作条
- 供应商官网链接写成占位文案或非真实网址
- 结果区只有展示态、没有清晰的编辑/审阅区分

## 8. 新增样式时的落点建议

为了保持一致性，新增样式时建议遵循下面的顺序：

1. **先复用 `ink-wash.css` 的变量**，不要直接写死颜色。
2. **优先复用 `app-chrome.css` 的共享组件**，例如按钮、toast、壳层、输入基线。
3. **页面特有样式再放到各自页面 CSS**，例如 `history.css`、`settings.css`。
4. **只有跨页面复用且已经稳定的模式，才回收进共享层。**

## 9. 维护原则

后续继续迭代前端时，风格层面统一遵守以下原则：

- 同一种交互，只保留一套视觉实现。
- 同一种状态，只保留一套主题映射规则。
- 同一种组件，只保留一套尺寸和动画基线。
- 页面的差异体现在业务内容，不体现在基础 UI 语言漂移。

一句话总结：**白石前端应始终像同一款桌面创作工具，而不是一组分别设计的静态页面。**
